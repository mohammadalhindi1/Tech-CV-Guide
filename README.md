# Tech CV Guide

A practical, evidence-informed guide to writing a clear technical CV for **IT, software engineering, cloud, DevOps, cybersecurity, data, and technical support roles**.

This repository is built for students, career changers, and junior professionals who need more than a polished template. It explains **what to include, why it matters, and how to turn real work into credible evidence**.

> A strong CV does not try to tell your whole life story. It helps a recruiter quickly understand the role you want, the skills you can prove, and the value you may bring.

## What this guide teaches

By the end, you should be able to:

- choose a clear target role instead of sending one generic CV everywhere;
- organize a concise technical CV around relevance and evidence;
- write a focused professional summary;
- group technical skills so they remain easy to scan;
- turn academic, personal, freelance, and open-source projects into credible experience;
- write bullet points that explain what you built, how you built it, and why it matters;
- tailor keywords honestly to a job description;
- avoid common ATS and formatting problems;
- create separate CV versions for different career paths.

## The recommended workflow

1. **Choose one target role.**  
   “Junior DevOps Engineer” is actionable. “Anything in IT” is not.

2. **Collect evidence.**  
   List projects, work, coursework, labs, certifications, tools, outcomes, links, and numbers you can defend.

3. **Select what is relevant.**  
   Keep the strongest evidence for the target role. A CV is a selection, not an archive.

4. **Draft in plain text first.**  
   Good wording matters more than visual decoration.

5. **Write evidence-based bullets.**  
   Use the pattern: **action + task + technology/context + result or purpose**.

6. **Tailor to the job description.**  
   Match relevant terminology without copying requirements or claiming skills you do not have.

7. **Format for fast reading.**  
   Use conventional headings, consistent dates, simple typography, and enough white space.

8. **Test the final file.**  
   Copy text from the PDF, check every link, proofread, and verify the filename.

## Recommended section order

A junior technical CV will usually work well in this order:

1. Name and contact information
2. Targeted professional summary
3. Technical skills
4. Projects
5. Experience
6. Education
7. Certifications, volunteering, or leadership when relevant

This is not a rigid law. If you have strong relevant professional experience, place **Experience** before **Projects**. If your projects prove more than your unrelated work history, place **Projects** first.

## One strong bullet is better than three vague bullets

**Weak**

> Worked with Docker and Kubernetes on a task API.

**Stronger**

> Containerized a FastAPI task-management service with Docker and deployed it to a local Kubernetes cluster using Deployment and Service manifests.

**Strongest when a real outcome is known**

> Containerized a FastAPI task-management service with a multi-stage Docker build, reducing the final image size by 38%, and deployed it to Minikube through Kubernetes Deployment and Service manifests.

Never invent a percentage, user count, performance improvement, or business result. When no metric exists, explain **scope, complexity, reliability, automation, or purpose**.

## What this repository does not promise

- No template can guarantee interviews.
- “ATS-friendly” does not mean there is one universal ATS score.
- Keyword matching cannot replace relevant evidence.
- AI-generated claims are not achievements.
- A beautiful design cannot rescue unclear or generic content.
- Different countries and employers use the terms *CV* and *resume* differently.

In this repository, **technical CV** means the concise job-application document commonly called a resume in the United States. Academic CVs, which include publications and extensive scholarly history, are outside the current scope.

## Quick rules

- Target one role or a closely related role family.
- Keep early-career CVs concise; one page is often a useful target, not an absolute law.
- Use standard headings such as **Experience**, **Projects**, **Education**, and **Technical Skills**.
- Prefer evidence over self-ratings such as “Linux: 90%.”
- List technologies you can discuss in an interview.
- Use reverse chronological order inside time-based sections.
- Keep date, punctuation, capitalization, and spacing styles consistent.
- Include GitHub or portfolio links only when the linked work is presentable.
- Save a master CV, then create a tailored copy for each application.
- Use a clear filename such as `Mohammad_Alhindi_DevOps_CV.pdf`.

## Common mistakes

- opening with an objective that only describes what the candidate wants;
- listing every technology ever encountered;
- using skill bars, stars, or unexplained percentages;
- describing project features without explaining personal contribution;
- repeating “responsible for” instead of using direct action verbs;
- adding fake metrics to make bullets sound impressive;
- using dense multi-column layouts without testing text extraction;
- including a photo, age, full street address, marital status, or national ID when not required;
- sending the same generic version to DevOps, support, and developer roles;
- linking to empty repositories or undocumented projects;
- allowing AI to write claims the candidate cannot explain.
