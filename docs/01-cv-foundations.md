# CV Foundations

## 1. The job of a technical CV

A technical CV is a decision-support document. Its purpose is to help a recruiter or hiring manager answer four questions quickly:

1. What role is this person targeting?
2. Do they show relevant technical ability?
3. What evidence supports their claims?
4. Is there enough reason to continue to an interview?

The CV does not need to contain everything you have done. It needs to contain the **most relevant and credible evidence** for a specific opportunity.

## 2. Start with a target, not a template

Before choosing fonts or writing a summary, define the target:

- target title: Junior DevOps Engineer;
- role family: cloud infrastructure and platform operations;
- strongest evidence: AWS project, Docker, Kubernetes, Linux, CI/CD;
- important gaps: Terraform and production monitoring;
- application market: Jordan, Europe, or remote roles.

A candidate targeting both DevOps and technical support should normally maintain two versions. The source material may overlap, but the emphasis must change.

| DevOps version emphasizes | Technical support version emphasizes |
|---|---|
| automation and deployment | troubleshooting and incident handling |
| Linux, cloud, containers, CI/CD | operating systems, networks, users, ticketing |
| infrastructure projects | diagnosis, communication, resolution |
| reliability and repeatability | response quality and service continuity |

## 3. Build an evidence inventory

Create a private master list before writing the CV. For every experience or project, record:

- problem or goal;
- your individual responsibility;
- tools and technologies;
- important decisions;
- scale or scope;
- test or validation method;
- outcome;
- repository, demo, certificate, or document that proves the work;
- facts and numbers you can verify.

This prevents two common problems: forgetting valuable work and inventing impressive-sounding details later.

### Evidence can come from more than employment

Junior candidates may use:

- academic projects;
- personal projects;
- labs and technical exercises;
- open-source contributions;
- freelance work;
- volunteering;
- student clubs;
- hackathons;
- certifications with hands-on work;
- relevant coursework.

Label each item honestly. A personal Kubernetes lab is valuable, but it is not production experience.

## 4. Relevance beats volume

Score each item from 0 to 2:

| Criterion | 0 | 1 | 2 |
|---|---:|---:|---:|
| Relevant to target role | no | partly | directly |
| Shows applied skill | no | somewhat | clearly |
| Has credible evidence | none | description | link/result |
| Recent enough to represent you | old | usable | current |
| Can explain confidently | no | mostly | yes |

Prioritize items with the highest total. Remove low-value entries before shrinking the font.

## 5. Length and layout

For a student or early-career candidate, one page is often a useful discipline because it forces prioritization. It is not a universal rule. Use a second page when the additional material is relevant and strong, not because spacing or wording is inefficient.

A readable document normally has:

- a simple hierarchy;
- consistent heading and date styles;
- comfortable margins and white space;
- body text that remains readable at normal zoom;
- bullets that usually fit within one or two lines;
- no unexplained visual ratings.

## 6. Accuracy is non-negotiable

You should be able to defend every item in an interview.

Do not:

- upgrade “basic exposure” into “expert”;
- call a tutorial an original project without explaining what you changed;
- claim team results as your individual result;
- use numbers you cannot reproduce;
- list tools only because they appear in the job description;
- publish secrets, private URLs, customer data, or internal screenshots.

## 7. Master CV and tailored CVs

Maintain two layers:

### Master CV

A private, longer source containing all valid experiences, bullets, projects, certifications, and skills.

### Tailored CV

A concise application version created from the master. It changes the ordering, summary, selected evidence, and terminology for one target role.

This is faster and safer than rewriting from memory for every application.

## Foundation test

Before drafting, you should be able to complete this sentence:

> I am targeting **[role]**, and my strongest evidence is **[project/experience]**, where I used **[relevant skills]** to **[result or purpose]**.

If that sentence is unclear, the CV will probably be unclear too.
