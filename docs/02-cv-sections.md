# Building Each CV Section

## 1. Header

Include:

- full professional name;
- city and country;
- professional email;
- phone number when appropriate;
- LinkedIn;
- GitHub or portfolio.

Example:

```text
MOHAMMAD ALHINDI
Amman, Jordan | email@example.com | +962 ...
linkedin.com/in/... | github.com/...
```

Avoid adding a full street address, national ID, marital status, date of birth, or photograph unless the employer, country, or application process specifically requires it. Public copies should use redacted contact details.

## 2. Professional summary

The summary should answer:

- Who are you professionally?
- What area do you target?
- Which relevant strengths can you support?
- What evidence makes the statement credible?

Useful structure:

> [Professional identity] with experience in [relevant area]. Built or supported [evidence] using [key technologies]. Strongest in [two or three strengths], targeting [role type].

Weak:

> Hardworking and passionate IT graduate seeking a challenging position to improve my skills.

Stronger:

> Cloud Computing graduate with hands-on experience building containerized services and deploying infrastructure with AWS, Docker, Kubernetes, Linux, and GitHub Actions. Developed projects covering REST APIs, CI/CD, networking, and cloud deployment, and is targeting junior DevOps or cloud engineering roles.

Do not fill the summary with unsupported adjectives such as *excellent*, *expert*, *innovative*, or *results-driven*.

## 3. Technical skills

Group skills by function. This helps a reader understand your toolbox without scanning a long mixed list.

```text
Cloud: AWS (EC2, S3, RDS, IAM), Azure fundamentals
DevOps: Docker, Docker Compose, Kubernetes, GitHub Actions
Systems: Linux, Bash, process management, permissions
Networking: TCP/IP, DNS, HTTP, subnetting, routing fundamentals
Programming: Python, C, C++, SQL
Tools: Git, GitHub, VS Code, Postman
```

Rules:

- include only skills you can explain;
- use specific services when relevant;
- do not list basic software such as Microsoft Word unless the role needs it;
- avoid stars, bars, and percentages;
- distinguish practical experience from fundamentals when wording would otherwise exaggerate;
- keep spelling and capitalization correct: Kubernetes, GitHub, PostgreSQL.

## 4. Projects

A project entry should include:

- project name;
- source or demo link;
- technologies;
- date when useful;
- two to four evidence-based bullets.

Example:

```text
Task Service API | GitHub
Python, FastAPI, Docker, Kubernetes, GitHub Actions
- Built REST endpoints for creating, reading, updating, and deleting tasks with Pydantic validation.
- Containerized the service with Docker and deployed it to Minikube using Kubernetes Deployment and Service manifests.
- Added automated checks through GitHub Actions to verify builds and core behavior on each push.
```

Explain your contribution when the project was completed by a team.

## 5. Experience

Use reverse chronological order. Each entry needs:

- job title;
- organization;
- location or remote;
- dates;
- relevant achievements and responsibilities.

A bullet should not be a copied job description. Focus on actions, scope, decisions, and outcomes.

If employment is unrelated, keep evidence that transfers to the target role: problem solving, customer support, documentation, reliability, teamwork, or leadership. Reduce details that do not help the application.

## 6. Education

For early-career candidates:

```text
BSc in Cloud Computing
Luminus Technical University College, Amman, Jordan | 2021–2025
Relevant coursework: Cloud Architecture, Linux Administration, Networking, Databases
```

Include GPA only when it is strong, requested, or beneficial in the target market. Do not list every course.

## 7. Certifications

Include the official certification name, issuing body, and year. Add the credential link or ID only when it is safe and useful.

Separate a completed certification from:

- course completion;
- training attendance;
- an exam currently in progress;
- a planned certification.

“Studying for AWS Solutions Architect” is not the same as holding the certification.

## 8. Optional sections

Add only when they strengthen the application:

- open-source contributions;
- volunteering;
- leadership;
- awards;
- technical writing;
- languages;
- relevant communities or events.

Interests can be useful when genuine and concise, but they should not replace technical evidence.

## Section ordering decision

Use this priority rule:

> Put the strongest relevant evidence where it will be seen earliest.

There is no benefit in placing Education above excellent relevant experience merely because a template used that order.
